# Phase 10.1: Fix WRAP_UP Bugs - Research

**Researched:** 2026-01-23
**Domain:** Cython entity handle debugging / State array offset management
**Confidence:** HIGH

## Summary

This phase fixes 3 critical bugs discovered during Phase 11 testing. All bugs are in the existing WRAP_UP implementation (phases/wrap_up.pyx) created in Phases 9 and 10.

**Root cause analysis:**
- **Bug 1 (FI cash zeroed)**: FI entity offset caching issue - FI cash becomes 0 after WRAP_UP regardless of purchases
- **Bug 2 (Player cash zeroed for players 1+)**: Player entity offset caching issue - players 1+ lose all cash after WRAP_UP cycle
- **Bug 3 (Infinite loop)**: Phase transition or FI purchase loop termination issue when no companies available

These are NOT logic bugs - they're Cython memory management bugs where entity handles (PLAYERS[], FI) have incorrect cached offsets pointing to wrong locations in the state array.

**Primary recommendation:** Audit all entity initialization and offset caching code. The Player and FI entity handles are global singletons that cache pointers into the state array. When these cached offsets are wrong, reads/writes go to incorrect memory locations.

## Architecture Patterns

### Entity Handle Pattern (Established in v1)

**What:** Global singleton entity objects (PLAYERS[], FI, COMPANIES[], etc.) that cache offsets into the state array.

**Key insight:** Entity handles are stateless wrappers. All state lives in GameState._data array. Entity methods take `GameState state` as first parameter.

**Initialization order (CRITICAL):**
1. GameState computes layout (num_players dependent)
2. Entity.initialize(state) caches offsets based on layout
3. Entity methods use cached offsets for fast array access

**Example from player.pyx:**
```cython
cpdef void initialize(self, GameState state):
    cdef StateLayout layout = state._layout
    cdef PlayerFieldOffsets fields = state._player_fields

    self._num_players = state._num_players
    self._base_offset = layout.players_offset + (self.player_id * layout.player_stride)

    # Cache absolute offsets for each field
    self._cash_offset = self._base_offset + fields.cash
    self._net_worth_offset = self._base_offset + fields.net_worth
    # ... etc
```

### State Array Layout (from VECTORS.md pattern)

Players are stored as contiguous blocks:
```
[Player 0 data block][Player 1 data block][Player 2 data block]...
```

Each player block contains:
```
[cash][net_worth][turn_order one-hot][is_auction_high_bidder][owned_companies][...]
```

**player_stride** = total size of one player's data block
**player_id** = 0-5 (fixed, never changes - PLAYERS[0] always has player_id=0)
**turn_order** = 0-5 (dynamic, changes during WRAP_UP reordering)

**CRITICAL DISTINCTION:**
- `player_id` identifies which entity handle (PLAYERS[0] vs PLAYERS[1])
- `turn_order` identifies position in play order (who goes first)
- Player reordering changes turn_order field, NOT player_id
- Cached offsets are based on player_id (fixed), not turn_order (dynamic)

### Offset Caching Invariant

**Rule:** Cached offsets MUST match the actual state layout.

**Violation patterns:**
1. **Missing field in cache**: Entity skips caching a field that exists in layout → all subsequent offsets wrong
2. **Wrong base offset**: Entity computes base_offset incorrectly → all offsets wrong
3. **Stale cache**: Entity initialized with old layout, state reallocated → all offsets invalid

**How to verify:**
```cython
# In entity.pyx, add assertions
cpdef void initialize(self, GameState state):
    # ... compute offsets ...

    # Verify: write sentinel, read back
    state._data[self._cash_offset] = 12345.0 / CASH_DIVISOR
    assert self.get_cash(state) == 12345, "Offset mismatch!"
```

## Common Pitfalls

### Pitfall 1: Forgetting to Cache a Field

**What goes wrong:** Entity.initialize() skips caching a field that exists in the state layout. All subsequent field offsets are off by that field's size.

**Example:**
```cython
# PlayerFieldOffsets in state.pyx has:
p.cash = 0
p.net_worth = 1
p.turn_order = 2
p.is_auction_high_bidder = 2 + num_players  # <-- EXISTS
p.owned_companies = 3 + num_players

# But Player.initialize() does:
self._cash_offset = base + fields.cash
self._net_worth_offset = base + fields.net_worth
self._turn_order_offset = base + fields.turn_order
# MISSING: is_auction_high_bidder !!!
self._owned_companies_offset = base + fields.owned_companies

# Result: owned_companies_offset is CORRECT (uses pre-computed fields.owned_companies)
# But if code later tries to access is_auction_high_bidder, it won't work
```

**Why it happens:** Copy-paste error when adding new fields, or field added to layout but not to entity class.

**How to avoid:**
1. Check that Player.pxd declares ALL fields in PlayerFieldOffsets
2. Check that Player.initialize() caches ALL declared fields
3. Run comprehensive tests that exercise ALL fields

**Warning signs:**
- Player 0 works, players 1+ broken → offset calculation bug
- Some fields preserved, others zeroed → partial offset corruption
- Bug appears after WRAP_UP but not during INVEST → phase-specific entity access pattern

### Pitfall 2: Entity Not Re-initialized After State Change

**What goes wrong:** GameState is modified (e.g., player reordering changes layout), but entity handles still have old cached offsets.

**Why it happens:** Entity handles are global singletons initialized once. If state layout changes, offsets become stale.

**Current codebase status:** This should NOT happen - layout is fixed after initialize_game(). Player reordering changes turn_order FIELD, not layout structure.

**How to detect:** Compare entity's cached offset with freshly computed offset:
```python
# In test
layout = state._layout
fields = state._player_fields
expected_offset = layout.players_offset + (player_id * layout.player_stride) + fields.cash
actual_offset = PLAYERS[player_id]._cash_offset
assert expected_offset == actual_offset, f"Stale cache! Expected {expected_offset}, got {actual_offset}"
```

### Pitfall 3: Infinite Loop in Phase Handler

**What goes wrong:** Phase handler completes without transitioning to a state that has available actions. GameDriver's auto-apply loop calls the handler repeatedly, hitting MAX_FORCED_ITERATIONS (100).

**Why it happens in WRAP_UP:**
- WRAP_UP transitions to ACQUISITION (sentinel -101)
- ACQUISITION transitions to INVEST (with actions)
- If transition fails, WRAP_UP runs again → infinite loop

**Possible causes:**
1. `turn_module.TURN.set_phase()` doesn't actually update phase
2. ACQUISITION handler has bug and doesn't transition
3. FI purchase while loop never terminates (e.g., is_for_auction always returns True)

**How to debug:**
```python
# Add prints to phase handlers
cdef int apply_wrap_up(GameState state) noexcept:
    printf("WRAP_UP: before FI purchases, phase=%d\n", state.get_phase())
    _process_fi_purchases(state)
    printf("WRAP_UP: after FI purchases, before transition\n")
    turn_module.TURN.set_phase(state, GamePhases.PHASE_ACQUISITION)
    printf("WRAP_UP: after transition, phase=%d\n", state.get_phase())
    return 0
```

**Warning signs:**
- ForcedActionLoopError in tests
- Specific test scenarios trigger it (e.g., "no companies available")
- Error occurs during auto-apply, not during direct action

## Open Questions

### Question 1: Is is_auction_high_bidder Field Missing from Player Cache?

**What we know:**
- PlayerFieldOffsets (core/state.pyx line 281) includes is_auction_high_bidder
- Player.pxd (line 73-83) declares cached offsets but doesn't list _is_auction_high_bidder_offset
- Player.initialize() (player.pyx line 260-267) doesn't cache is_auction_high_bidder offset

**What's unclear:**
- Is this intentional (field unused so not cached)?
- Or is this the bug (skipped field causes subsequent offsets to be wrong)?

**Recommendation:**
- Audit player.pyx to see if is_auction_high_bidder is accessed anywhere
- If YES: this is the bug - add it to cached offsets
- If NO: verify that fields.owned_companies already accounts for it (it should)

**Most likely:** This is NOT the bug because Player.initialize() uses `fields.owned_companies` which is pre-computed to include all prior fields. The bug is elsewhere.

### Question 2: Why Does Bug Only Affect Players 1+?

**What we know:**
- Player 0's cash is preserved correctly
- Players 1+ have cash become 0
- All players use same entity class (Player)
- Offset calculation is: `base_offset = layout.players_offset + (player_id * layout.player_stride)`

**What's unclear:**
- Is player_stride calculated incorrectly?
- Is there an off-by-one error in the loop that zeros cash?
- Is the bug in ACQUISITION.clear_roundtrip_tracking()?

**Recommendation:**
- Add test that sets ALL player fields to unique values, then checks ALL fields after WRAP_UP
- This will reveal exactly which fields are corrupted and which are preserved
- Pattern of corruption will reveal the offset bug location

### Question 3: Is FI Cash Bug Related to Player Cash Bug?

**What we know:**
- Both FI and Player cash become 0 after WRAP_UP
- FI entity uses same pattern as Player entity (cached offsets)
- FI.initialize() is simpler (only 2 fields: cash, owned_companies)

**What's unclear:**
- Is there a common bug in both entities?
- Or are these separate bugs with same symptom?

**Recommendation:**
- Fix one bug, re-test the other
- If both bugs persist independently, they're separate issues
- If fixing one fixes both, there's a common root cause (likely in state array write)

### Question 4: What Causes Infinite Loop When No Companies Available?

**What we know:**
- Test clears all companies with `remove_from_game()`
- FI purchase loop should terminate immediately (no companies for auction)
- Instead, ForcedActionLoopError occurs (100 iterations)

**What's unclear:**
- Does `_find_cheapest_affordable_available()` return -1 correctly?
- Does `_process_fi_purchases()` while loop terminate correctly?
- Does phase transition to ACQUISITION actually happen?
- Does ACQUISITION transition to INVEST?

**Recommendation:**
- Add printf debugging to trace phase transitions
- Add assertion in FI purchase loop: `assert iteration_count < 10, "FI loop stuck"`
- Check if the issue is FI loop or phase transition
